\section{Introduzione}
\subsection{Scopo del documento}
Il presente documento si pone l'obiettivo di definire in modo esaustivo la \textbf{Specifica Tecnica} del progetto, fornendo una guida dettagliata alle scelte architetturali e implementative adottate. Esso rappresenta il ponte logico tra i requisiti iniziali e l'effettiva realizzazione del software, garantendo che ogni componente sia integrata in modo coerente e scalabile.

Il documento approfondisce l'evoluzione del sistema rispetto al \textbf{Proof of Concept (PoC)} iniziale, delineando una struttura robusta e orientata alla manutenibilità a lungo termine.

Nello specifico, questo documento si propone di:
\begin{itemize}
  \item \textbf{Delineare l'architettura di sistema}: fornendo una panoramica delle macro-componenti e del flusso di dati tra di esse;
  \item \textbf{Dettagliare le scelte tecnologiche}: giustificando l'adozione di specifici framework, librerie e strumenti di sviluppo;
  \item \textbf{Illustrare i design pattern}: descrivendo gli schemi logici e le best practice utilizzate per risolvere problematiche ricorrenti e ottimizzare la struttura del codice;
  \item \textbf{Definire i processi di deployment}: spiegando come le diverse componenti vengono distribuite, configurate e rese operative nell'ambiente di produzione;
  \item \textbf{Fornire una base per la manutenzione}: offrendo la documentazione necessaria affinché futuri sviluppatori possano comprendere ed estendere il sistema con facilità.
\end{itemize}
\subsection{Glossario}
Al fine di prevenire ambiguità e garantire una comunicazione uniforme e precisa tra i membri del gruppo e i committenti, è stato redatto un Glossario\textsuperscript{G} apposito.
Per facilitare la lettura del presente documento, i termini tecnici o dotati di un significato specifico all'interno del dominio di progetto sono contrassegnati da una lettera «G» posta in apice (es. Parola\textsuperscript{G}).
La definizione estesa di tali termini è reperibile nel documento citato tra i riferimenti informativi.
\subsection{Scopo del prodotto}
% Preso dall'AdR
Il sistema CodeGuardian è progettato per condurre analisi approfondite sulla qualità di repository GitHub\textsuperscript{G}, garantendo un controllo rigoroso sulla gestione degli accessi e sull'autorizzazione all'avvio delle procedure di scansione.
\\
La piattaforma si propone come uno strumento strategico per team di sviluppo multidisciplinari, consentendo il monitoraggio centralizzato dello stato del software e la generazione di metriche aggregate su molteplici progetti analizzati.

\section{Tecnologie}
Questa sezione descrive gli strumenti e le tecnologie adottati per lo sviluppo, il testing e il deployment del software relativo al progetto CodeGuardian. In particolare, vengono illustrati il linguaggio di programmazione, le librerie, i framework e le infrastrutture utilizzate.

% =============================================
% 2.1 LINGUAGGI DI PROGRAMMAZIONE
% =============================================
\subsection{Linguaggi di programmazione}

\subsubsection{TypeScript}
TypeScript è un linguaggio di programmazione open-source sviluppato da Microsoft. Si tratta di un superset di JavaScript, che introduce la tipizzazione statica, consentendo di rilevare errori di programmazione in fase di sviluppo, ridurre il numero di bug e migliorare la qualità del codice.
\begin{itemize}
  \item \textbf{Versione:} 5.7.3
  \item \textbf{Documentazione:} \url{https://www.typescriptlang.org/docs/} [visitato il 21/04/2026]
\end{itemize}
Nel contesto del progetto CodeGuardian, TypeScript è il linguaggio principale utilizzato per lo sviluppo sia del frontend che del backend. La tipizzazione statica è particolarmente utile in un progetto che coinvolge strutture dati complesse, come i report di analisi del codice e i grafi di scansione generati dall'agente AI\@.

Per il frontend si utilizza TSX, un'estensione di TypeScript che permette di definire la struttura dell'interfaccia utente in modo più leggibile e dichiarativo, integrando HTML direttamente nel codice TypeScript.

% =============================================
% 2.2 FORMATO DEI DATI
% =============================================
\subsection{Formato dei dati}

\subsubsection{JSON}
JSON (JavaScript Object Notation) è un formato di scambio dati leggero e indipendente dal linguaggio di programmazione. È ampiamente utilizzato nello sviluppo web e nello scambio di dati tra applicazioni. JSON è basato su due strutture di dati:
\begin{itemize}
  \item \textbf{Oggetti:} insiemi di coppie chiave-valore;
  \item \textbf{Array:} liste ordinate di valori.
\end{itemize}
Nel progetto CodeGuardian, JSON viene utilizzato per la trasmissione dei dati tra frontend e backend (payload delle API REST\textsuperscript{G}), per la serializzazione dei risultati di analisi, nonché per la configurazione dell'applicazione e la memorizzazione dei dati in MongoDB\@.

% =============================================
% 2.3 FRAMEWORK
% =============================================
\subsection{Framework}

\subsubsection{NestJS}
NestJS è un framework per Node.js\textsuperscript{G} che offre un'architettura modulare e scalabile per la creazione di applicazioni lato server. Usa Express.js come HTTP adapter di default e fornisce funzionalità aggiuntive come dependency injection\textsuperscript{G}, middleware e decorator. NestJS consente di organizzare il codice in moduli riutilizzabili e di definire facilmente controller, servizi e provider.
\begin{itemize}
  \item \textbf{Versione:} 11.0.1
  \item \textbf{Documentazione:} \url{https://docs.nestjs.com/} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, NestJS è utilizzato per creare il backend. In particolare, viene impiegato per definire i controller che gestiscono le richieste HTTP provenienti dal frontend, i servizi che implementano la business logic (gestione workspace\textsuperscript{G}, membership\textsuperscript{G}, scansioni), e i provider per l'iniezione delle dipendenze. L'architettura del backend è organizzata in sette moduli principali: \texttt{WorkspaceModule}, \texttt{MembershipModule}, \texttt{RepositoryModule}, \texttt{ScanModule}, \texttt{ReportModule}, \texttt{AnalysisModule} e \texttt{GitHubModule}.

\subsubsection{LangGraph}
LangGraph è un framework open-source per l'orchestrazione di agenti AI\textsuperscript{G}, sviluppato da LangChain Inc. Consente di modellare workflow complessi come grafi a stati, dove ogni nodo rappresenta un passo di ragionamento o un'azione, e ogni arco il flusso di controllo tra i passi. LangGraph fornisce funzionalità di gestione dello stato, checkpointing, streaming token-by-token e supporto per cicli e ramificazioni condizionali.
\begin{itemize}
  \item \textbf{Versione:} 1.2.6
  \item \textbf{Documentazione:} \url{https://langchain-ai.github.io/langgraph/} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, LangGraph è il componente centrale dell'agente di analisi del codice. Il grafo implementato definisce il flusso di scansione dei repository GitHub\textsuperscript{G}: dalla clonazione del repository, all'analisi dei file, fino alla generazione del report finale. La struttura a grafo consente di gestire ramificazioni condizionali basate sul tipo di analisi richiesta e permette di integrare verifiche intermedie e retry automatici in caso di errore.

\subsection{Librerie}

% --- 2.4.1 Librerie Backend ---
\subsubsection{Librerie backend}

\subsubsubsection{LangChain}
LangChain è una libreria open-source che fornisce componenti riutilizzabili per l'integrazione di Large Language Model (LLM)\textsuperscript{G} in applicazioni software. Offre astrazioni per la gestione dei prompt, l'invocazione dei modelli, l'integrazione con tool esterni e la gestione della memoria conversazionale.
\begin{itemize}
  \item \textbf{Versione:} 1.1.36
  \item \textbf{Documentazione:} \url{https://js.langchain.com/docs/} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, LangChain è utilizzato in combinazione con LangGraph per la costruzione dell'agente di analisi. Fornisce le astrazioni per l'invocazione dei modelli LLM e l'integrazione con i tool di analisi del codice sorgente.

\subsubsubsection{Axios}
Axios è un client HTTP basato su Promise per Node.js e browser. È isomorfo, cioè può essere eseguito sia nel browser che in Node.js utilizzando lo stesso codice sorgente. Lato server, Axios utilizza il modulo \texttt{http} nativo di Node.js, mentre lato client usa \texttt{XMLHttpRequest}.
\begin{itemize}
  \item \textbf{Versione:} 1.14
  \item \textbf{Documentazione:} \url{https://axios-http.com/docs/intro} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, Axios viene utilizzato lato frontend per effettuare richieste HTTP al backend (interrogazione delle API REST per workspace, scansioni e report) e lato backend per comunicare con servizi esterni, come le API di GitHub.

\subsubsubsection{dotenv}
dotenv è una libreria che consente di caricare variabili d'ambiente da un file \texttt{.env} nell'oggetto \texttt{process.env} di Node.js. Permette di separare la configurazione dal codice sorgente, seguendo le best practice della metodologia Twelve-Factor App.
\begin{itemize}
  \item \textbf{Versione:} 17.3
  \item \textbf{Documentazione:} \url{https://github.com/motdotla/dotenv} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, dotenv è utilizzato per gestire in modo sicuro le credenziali sensibili (chiavi API dei servizi LLM, credenziali MongoDB, configurazioni AWS) sia in ambiente di sviluppo che di testing, evitando di esporre dati riservati nel codice sorgente.

\subsubsubsection{isomorphic-git}
isomorphic-git è una reimplementazione completa di Git in puro JavaScript, compatibile sia con Node.js che con ambienti browser. Consente di eseguire operazioni Git (clone, checkout, log, status) senza dipendenze da moduli nativi C++, garantendo piena interoperabilità con l'implementazione canonica di Git.
\begin{itemize}
  \item \textbf{Versione:} 1.37
  \item \textbf{Documentazione:} \url{https://isomorphic-git.org/} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, isomorphic-git è una componente fondamentale del modulo \texttt{GitHubModule}. Viene utilizzato per clonare i repository GitHub degli utenti direttamente nel backend, consentendo all'agente AI di analizzare il codice sorgente localmente senza dipendere da strumenti CLI esterni. Ciò garantisce portabilità e compatibilità con ambienti containerizzati.

\subsubsubsection{Mongoose}
Mongoose è una libreria ODM (Object Data Modeling) per MongoDB e Node.js. Fornisce una soluzione basata su schema per modellare i dati dell'applicazione, includendo validazione built-in, casting dei tipi, costruzione di query e hook per la business logic.
\begin{itemize}
  \item \textbf{Versione:} 9.3
  \item \textbf{Documentazione:} \url{https://mongoosejs.com/docs/} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, Mongoose è utilizzato come layer di accesso ai dati per MongoDB\@. Tutti i modelli del dominio applicativo (workspace, membership, repository, scan, report) sono definiti tramite schema Mongoose, garantendo la validazione dei dati a livello applicativo e la coerenza delle operazioni CRUD\@.

\subsubsubsection{Passport.js}
Passport.js è un popolare middleware di autenticazione per Node.js, progettato per essere estremamente flessibile e modulare. Il suo punto di forza principale risiede nell'utilizzo delle cosiddette strategie, che permettono agli sviluppatori di integrare facilmente oltre 500 metodi di accesso diversi all'interno di un'applicazione
\begin{itemize}
  \item \textbf{Versione:} 0.7.0
  \item \textbf{Documentazione:} \url{https://www.passportjs.org/docs/} [visitato il 21/04/2026]
\end{itemize}
Nel progetto Passport.js viene utilizzato per gestire l'autenticazione di ogni richiesta http, controllando che l'access token ottenuto facendo il login tramite AWS Cognito sia valido


% --- 2.4.2 Librerie Frontend ---
\subsubsection{Librerie frontend}

\subsubsubsection{Tailwind CSS}
Tailwind CSS è un framework CSS utility-first che permette di costruire interfacce utente
direttamente nel markup, componendo classi di basso livello invece di scrivere CSS personalizzato.
A differenza dei framework tradizionali basati su componenti predefiniti, Tailwind fornisce
primitive atomiche altamente componibili che offrono pieno controllo sul design.
\begin{itemize}
  \item \textbf{Versione:} 4.1
  \item \textbf{Documentazione:} \url{https://tailwindcss.com/docs} [visitato il 21/04/2026]
\end{itemize}
Nel progetto CodeGuardian, Tailwind CSS è utilizzato come sistema di stile principale per
l'intera interfaccia frontend. Ogni componente React è stilizzato tramite classi utility,

\section{Architettura di deployment}
Per la distribuzione dell'applicazione nell'ambiente di produzione AWS, si è optato per un'architettura monolitica containerizzata. In questo modello, tutte le componenti del backend (gestione utenti, analisi del codice, generazione report) sono incluse in un'unica immagine Docker e distribuite come un singolo servizio su Amazon ECS.\\ \\
Motivazioni della scelta:
\begin{itemize}
  \item Proporzionalità rispetto alla scala del progetto: per un sistema delle dimensioni di CodeGuardian, l'adozione di un'architettura a microservizi risulterebbe ingente da implementare e nettamente sovradimensionata. Il passaggio a sistemi distribuiti introdurrebbe una complessità accidentale non giustificata dal carico di lavoro previsto;
  \item Semplificazione del flusso DevOps: la gestione di un unico artefatto riduce drasticamente l'overhead operativo relativo alla configurazione dei cluster ECS e dei bilanciatori di carico;
  \item Efficienza nelle operazioni I/O: poiché l'analisi del codice richiede la clonazione locale dei repository, la presenza di tutti i moduli nello stesso ambiente di esecuzione elimina la latenza di rete che si verificherebbe in un'architettura a microservizi;
\end{itemize}

\section{Architettura logica}
\subsection{Frontend}
\subsubsection{Introduzione}
L'architettura del frontend di \textit{CodeGuardian} adotta il pattern \textbf{MVVM} (Model-View-ViewModel), che separa l'applicazione in tre layer con dipendenze strettamente unidirezionali:

\begin{center}
  \textbf{View} $\longrightarrow$ \textbf{ViewModel} $\longrightarrow$ \textbf{Model} $\longrightarrow$ \texttt{apiClient}
\end{center}
\textbf{View}: componenti React responsabili della presentazione e dell'interazione con l'utente. Si suddividono in \textit{componenti pagina}, che compongono la struttura delle schermate e applicano le guard di autenticazione (\texttt{ProtectedRoute}), e \textit{componenti di presentazione}, che ricevono dati tramite props o invocando hook del layer ViewModel e si occupano esclusivamente del rendering;
\newline
\textbf{ViewModel}: custom hook React che incapsulano la logica di stato e orchestrano le operazioni di lettura e scrittura. Utilizzano \texttt{useQuery} e \texttt{useMutation} di TanStack Query per gestire rispettivamente il recupero dati con caching automatico e le operazioni di modifica con invalidazione reattiva della cache, eccetto in casi specifici in cui la struttura gerarchica dei provider React non rende disponibile il client TanStack Query, dove si ricorre a \texttt{useState}/\texttt{useEffect};
\newline
\textbf{Model}: il layer di dominio è costituito dai DTO tipizzati che rappresentano le entità applicative e dai repository che ne gestiscono il recupero e la persistenza. I repository implementano interfacce segregate per responsabilità secondo il principio ISP: ad esempio, le operazioni sui repository di un workspace sono suddivise in \texttt{IGetRepositoriesRepository}, \\ \texttt{IAddRepositoryRepository}, \texttt{IRemoveRepositoryRepository} e \texttt{IUpdateTokenRepository}, quattro interfacce distinte invece di un'unica interfaccia monolitica. La comunicazione HTTP con il backend avviene tramite il modulo condiviso \texttt{apiClient}, che espone funzioni tipizzate per i verbi HTTP (\texttt{apiGet}, \texttt{apiPost}, \texttt{apiPut}, \texttt{apiPatch}, \texttt{apiDelete}). Ogni repository viene esportato come \textit{istanza singleton} tipizzata sull'interfaccia, impedendo ai consumer di accedere a metodi non previsti dal contratto e consentendo al contempo la sostituzione con implementazioni mock nei test unitari senza modificare il codice di produzione.
\newline
\newline
Nessun layer inferiore ha conoscenza dei layer superiori: i repository non conoscono né gli hook che li consumano né i componenti che renderizzano i dati, ma ricevono parametri primitivi e restituiscono DTO tipizzati. I contratti tra i layer sono definiti da interfacce TypeScript suddivise in due categorie: \textit{interfacce Model}, che dichiarano i metodi esposti dai repository, e \textit{interfacce ViewModel}, che dichiarano le proprietà e le azioni restituite dagli hook alla View. Dove rilevante per la testabilità, gli hook accettano l'implementazione del repository come parametro opzionale con valore di default, consentendo la sostituzione con mock nei test unitari tramite dependency injection; per gli hook basati su TanStack Query (\texttt{useMutation}, \texttt{useQuery}), il mocking avviene invece a livello di modulo.

Il routing è gestito da \textbf{TanStack Router} con tipizzazione statica dei parametri di rotta. Le rotte protette sono avvolte dal componente \texttt{ProtectedRoute}, che verifica lo stato di autenticazione tramite \texttt{useAuthContext} e, in caso di utente non autenticato, reindirizza automaticamente al flusso di login preservando il percorso richiesto per consentire il ritorno alla destinazione originale dopo l'accesso.

\subsubsection{Design pattern}




\subsubsubsection{Repository}
Il pattern \textbf{Repository} è applicato a ogni accesso al backend: le chiamate HTTP e la serializzazione dei dati sono incapsulate in classi dedicate che espongono ai consumer soltanto metodi di dominio, nascondendo i dettagli del protocollo di trasporto e dei percorsi degli endpoint. Gli hook del layer ViewModel invocano questi metodi senza conoscere la sorgente effettiva dei dati, ottenendo così una chiara separazione tra la logica applicativa e l'infrastruttura di comunicazione.


\subsubsubsection{Dependency Injection}
Il pattern \textbf{Dependency Injection} ha l'obiettivo di minimizzare il grado di dipendenza fra le parti di un sistema, fornendo tecniche per passare le dipendenze a una classe in modo che l'accoppiamento fra le parti risulti il più lasco possibile. Nel frontend questo pattern è realizzato negli hook che necessitano di interagire con i repository, tramite un parametro opzionale che accetta l'implementazione del repository e che assume come valore di default il singleton di produzione. In questo modo l'hook resta indipendente rispetto all'implementazione concreta con cui verrà utilizzato e può ricevere dall'esterno una versione alternativa qualora il contesto lo richieda.



\subsubsection{Infrastruttura condivisa}

Prima di descrivere le singole feature si descrivono i principali componenti trasversali
utilizzati dall'intero frontend. 

\subsubsubsection{apiClient}
\includegraphics[width=1\textwidth]{../Assets/specifica_tecnica/Feature-apiClient.png}


Il modulo \texttt{apiClient} è il punto di accesso unico per tutte le chiamate
HTTP verso il backend applicativo. Ogni repository concreto del layer Model
delega a questo modulo l'esecuzione delle richieste, evitando la duplicazione
della logica di autenticazione e di gestione degli errori. Il modulo espone
cinque funzioni tipizzate, una per ciascun verbo HTTP utilizzato dal sistema:
\begin{itemize}
  \item \texttt{apiGet<T>(path: string): Promise<T>} — esegue una richiesta
        \texttt{GET} all'endpoint specificato;
  \item \texttt{apiPost<T>(path: string, body?: unknown): Promise<T>} — esegue
        una richiesta \texttt{POST}, serializzando opzionalmente il body come
        JSON;
  \item \texttt{apiPut<T>(path: string, body?: unknown): Promise<T>} — esegue
        una richiesta \texttt{PUT} con le stesse semantiche di \texttt{apiPost};
  \item \texttt{apiPatch<T>(path: string, body?: unknown): Promise<T>} —
        esegue una richiesta \texttt{PATCH};
  \item \texttt{apiDelete<T>(path: string): Promise<T>} — esegue una richiesta
        \texttt{DELETE}.
\end{itemize}

Ogni funzione aggiunge automaticamente un header \texttt{Authorization: Bearer
<token>} alla richiesta, recuperando l'\texttt{accessToken} dalla sessione
Amplify corrente tramite \texttt{fetchAuthSession}. Questo garantisce che tutte
le chiamate al backend siano autenticate senza richiedere ai repository di
gestire esplicitamente i token. Fa eccezione la registrazione iniziale
dell'utente, gestita da \texttt{AuthRepository}, che non utilizza
\texttt{apiClient} perché richiede l'\texttt{idToken} invece
dell'\texttt{accessToken} per accedere agli \textit{identity claims} necessari
al backend.

Al termine di ogni richiesta, il modulo esamina lo stato HTTP della risposta
tramite la funzione interna \texttt{handleResponse}. Se la risposta indica
successo, il corpo viene deserializzato come JSON e restituito; se il corpo
è vuoto viene restituito \texttt{undefined}. Se la risposta indica un errore,
viene sollevata un'istanza di \texttt{ApiError} contenente lo status HTTP e un
messaggio estratto dal corpo della risposta quando disponibile, o un messaggio
di default altrimenti.

\subsubsubsection{ApiError}

\textbf{ApiError} è la classe di errore utilizzata in tutto il frontend per
rappresentare i fallimenti delle chiamate HTTP. Estende la classe nativa
\texttt{Error} aggiungendo il campo pubblico \texttt{readonly status: number},
che espone il codice di stato HTTP della risposta fallita. Questa
caratterizzazione consente ai repository di implementare logiche di gestione
differenziate in base al tipo di errore: un esempio è
\texttt{GetBranchesRepository}, che tratta gli errori \texttt{404} e
\texttt{403} come assenza di branch (restituendo un array vuoto) e propaga
invece tutti gli altri errori al chiamante.

\subsubsubsection{Componenti di presentazione condivisi}

\textbf{RepositoryCard} è il componente di presentazione che visualizza un
repository all'interno di una card. Viene utilizzato dalla feature
\textit{WorkspaceRepository} all'interno di \texttt{RepositoryItem} per
rappresentare ciascun repository nella lista del workspace. Riceve le seguenti
props:
\begin{itemize}
  \item \texttt{name: string} — nome del repository;
  \item \texttt{ownerName: string} — username del proprietario;
  \item \texttt{dateScan?: string} — data dell'ultima scansione, opzionale;
  \item \texttt{documentationScore?: number} — punteggio della documentazione,
        opzionale;
  \item \texttt{codeCoverage?: number} — percentuale di code coverage,
        opzionale;
  \item \texttt{cvss?: number} — punteggio di sicurezza CVSS, opzionale;
  \item \texttt{onRemove?: () => void} — callback opzionale invocata al click
        sul pulsante di rimozione;
  \item \texttt{isRemoving?: boolean} — indica se la rimozione è in corso,
        disabilitando il pulsante.
\end{itemize}
Il componente mostra il nome del repository, il suo proprietario, le tre
metriche principali tramite barre di progresso colorate in base alla soglia
raggiunta e la data dell'ultima scansione formattata.

\paragraph{ScoreBar} è il componente di presentazione che visualizza un punteggio
numerico su scala da 0 a 10 tramite una barra di progresso colorata. Viene
utilizzato dalla feature \textit{Report} all'interno di \texttt{DocsSection} e
\texttt{SecuritySection} per visualizzare i voti delle singole categorie di
analisi. Riceve le seguenti props:
\begin{itemize}
  \item \texttt{label: string} — etichetta descrittiva del punteggio;
  \item \texttt{value: number} — valore del punteggio su scala da 0 a 10.
\end{itemize}
Il colore della barra varia in base al valore: verde per punteggi superiori o
uguali a 7, giallo per punteggi compresi tra 5 e 7, rosso per punteggi
inferiori a 5.

\subsubsection{Feature}
Il frontend è organizzato in feature indipendenti, ciascuna corrispondente a un'area funzionale dell'applicazione. Ogni feature replica internamente la struttura a tre layer dell'architettura MVVM, contenendo i propri componenti View, i propri hook ViewModel, i propri repository Model, le relative interfacce e i tipi di dominio.


\subsubsubsection{Auth}
\begin{figure}[H]
    \centering
        \includegraphics[width=1\textwidth]{../Assets/specifica_tecnica/Feature-auth.png}    
        \caption{UML feature Auth}
    \label{fig:feature Auth}
\end{figure}



La feature \textit{Auth} gestisce l'autenticazione degli utenti tramite Amazon Cognito con il
flusso OAuth 2.0. Al primo accesso, l'utente viene reindirizzato alla pagina di login di
Cognito; al ritorno, la \texttt{CallbackPage} verifica la sessione, registra l'utente nel
database applicativo se necessario e lo reindirizza alla destinazione originale. Le route
protette vengono sorvegliate dal componente \texttt{ProtectedRoute}, che reindirizza
automaticamente a Cognito gli utenti non autenticati preservando il percorso richiesto.
Lo stato di autenticazione è reso disponibile globalmente tramite il context \texttt{AuthContext},
popolato dall'\texttt{AuthProvider} e consumato dagli altri componenti tramite all'hook
\texttt{useAuthContext}.

\subsubsubsubsection{Tipi e DTO}

\textbf{AuthState} è il tipo che modella lo stato di autenticazione dell'utente. Espone i
seguenti campi:
\begin{itemize}
  \item \texttt{isAuthenticated: boolean} — indica se l'utente è attualmente autenticato;
  \item \texttt{isLoading: boolean} — indica se la verifica della sessione è in corso;
  \item \texttt{user: \{ username: string \} | null} — dati dell'utente autenticato, oppure
        \texttt{null} se non autenticato.
\end{itemize}

\subsubsubsubsection{Componenti View}

\textbf{AuthProvider} è il componente che inizializza e fornisce il context di autenticazione
all'albero dei componenti figli. Riceve la prop \texttt{children: ReactNode}, istanzia
internamente l'hook \texttt{useAuth} e ne passa il risultato come valore del
\texttt{AuthContext.Provider}. Deve essere posizionato in cima all'albero dei componenti per
rendere lo stato di autenticazione disponibile a tutti i discendenti.

\textbf{ProtectedRoute} è il componente che protegge le route che richiedono autenticazione.
Riceve la prop \texttt{children: ReactNode} e consuma lo stato di autenticazione tramite
\texttt{useAuthContext}. Se l'utente non è autenticato e il caricamento è terminato, invoca
\texttt{login} passando il percorso corrente come \texttt{redirectTo}, in modo che dopo il
login l'utente venga reindirizzato alla destinazione originale. Durante il caricamento mostra
un messaggio testuale; se l'utente non è autenticato non renderizza nulla; se autenticato
renderizza i componenti figli.

\textbf{CallbackPage} è il componente che gestisce il ritorno dal flusso OAuth di Cognito.
Non riceve props e delega interamente la logica all'hook \texttt{useCallbackPage}. Mostra
un messaggio testuale di attesa durante l'elaborazione del callback.

\subsubsubsubsection{Hook}

\textbf{useAuth} è il custom hook principale della feature, responsabile della gestione dello
stato di autenticazione. Implementa l'interfaccia \texttt{IAuthViewModel} e gestisce
internamente lo stato tramite \texttt{useState} e \texttt{useEffect}. Accetta opzionalmente
un'implementazione di \texttt{IAuthApi} per facilitare il testing tramite dependency injection.
Espone i seguenti membri:
\begin{itemize}
  \item \texttt{isAuthenticated: boolean} — indica se l'utente è autenticato;
  \item \texttt{isLoading: boolean} — indica se la verifica della sessione è in corso;
  \item \texttt{user: \{ username: string \} | null} — dati dell'utente autenticato;
  \item \texttt{login(redirectTo?: string): Promise<void>} — salva opzionalmente il percorso
        di destinazione in \texttt{sessionStorage} e avvia il flusso di login con redirect
        verso Cognito;
  \item \texttt{logout(): Promise<void>} — esegue il logout da Cognito e reindirizza
        l'utente alla pagina principale tramite \texttt{window.location.href};
  \item \texttt{checkAuth(): Promise<void>} — verifica la sessione corrente invocando
        \texttt{fetchCurrentUser} e \texttt{fetchSession}; aggiorna lo stato di autenticazione
        in base al risultato.
\end{itemize}

\textbf{useAuthContext} è il custom hook che consente ai componenti di consumare il context
di autenticazione. Invoca \texttt{useContext} su \texttt{AuthContext} e lancia un errore
esplicito se utilizzato al di fuori di un \texttt{AuthProvider}, garantendo un utilizzo
corretto del context.

\textbf{useCallbackPage} è il custom hook che gestisce la logica del callback OAuth.
Implementa l'interfaccia \texttt{ICallbackPageViewModel} e orchestra il flusso post-login
tramite \texttt{useEffect}. Al montaggio esegue in sequenza le seguenti operazioni: verifica
che l'utente sia presente in Cognito tramite \texttt{fetchCurrentUser}; tenta la registrazione
nel database applicativo tramite \texttt{useRegister}; in caso di successo legge il percorso
salvato in \texttt{sessionStorage} e naviga alla destinazione; in caso di fallimento della
registrazione rimuove l'utente da Cognito tramite \texttt{removeCurrentUser} e naviga alla
home con un parametro di errore. Accetta opzionalmente un'implementazione di \texttt{IAuthApi}
per il testing.

\textbf{useRegister} è il custom hook responsabile della registrazione dell'utente nel
database. Implementa l'interfaccia \texttt{IRegisterViewModel} e delega
l'operazione all'implementazione di \texttt{IAuthRepository} ricevuta per dependency injection.
Espone il membro:
\begin{itemize}
  \item \texttt{register(): Promise<void>} — invoca il metodo \texttt{register} del repository
        per registrare l'utente autenticato nel backend.
\end{itemize}

\textbf{IAuthViewModel} è l'interfaccia che definisce il contratto esposto da \texttt{useAuth}
verso il layer View. Estende \texttt{AuthState} aggiungendo i membri \texttt{login},
\texttt{logout} e \texttt{checkAuth}, con le stesse semantiche descritte per l'hook.

\textbf{ICallbackPageViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useCallbackPage}. Nel codice attuale è un'interfaccia vuota, in quanto il componente
\texttt{CallbackPage} non necessita di dati esposti dall'hook.

\textbf{IRegisterViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useRegister}. Dichiara il solo membro \texttt{register(): Promise<void>}.

\subsubsubsubsection{Repository e Service}

\textbf{IAuthApi} è l'interfaccia che definisce il contratto del service di autenticazione
verso Amazon Cognito tramite AWS Amplify. Dichiara i seguenti metodi:
\begin{itemize}
  \item \texttt{fetchCurrentUser(): Promise<\{ username: string \}>} — recupera l'utente
        attualmente autenticato dalla sessione Amplify;
  \item \texttt{fetchSession(): Promise<AuthSession>} — recupera la sessione corrente,
        inclusi i token JWT;
  \item \texttt{signIn(redirectUrl?: string): Promise<void>} — salva opzionalmente il percorso
        di destinazione e avvia il flusso OAuth con redirect verso Cognito;
  \item \texttt{logOut(): Promise<void>} — esegue il logout da Amplify e reindirizza
        l'utente all'endpoint di logout di Cognito;
  \item \texttt{removeCurrentUser(): Promise<void>} — elimina l'utente corrente da Cognito,
        utilizzato in caso di fallimento della registrazione nel database applicativo.
\end{itemize}

\textbf{AuthApi} è il service concreto che implementa \texttt{IAuthApi} wrappando le funzioni
di AWS Amplify (\texttt{getCurrentUser}, \texttt{fetchAuthSession}, \texttt{signInWithRedirect},
\texttt{signOut}, \texttt{deleteUser}). Il metodo \texttt{logOut} esegue prima il logout da
Amplify, poi reindirizza all'endpoint di logout del dominio Cognito configurato per invalidare
la sessione lato provider.

\textbf{IAuthRepository} è l'interfaccia che definisce il contratto per la registrazione
dell'utente nel database applicativo. Dichiara il metodo:
\begin{itemize}
  \item \texttt{register(): Promise<void>} — registra l'utente autenticato nel backend
        applicativo tramite una richiesta HTTP autenticata con \texttt{idToken}.
\end{itemize}

\textbf{AuthRepository} è la classe concreta che implementa \texttt{IAuthRepository}.
Recupera l'\texttt{idToken} dalla sessione Amplify corrente e invia una richiesta
\texttt{POST} all'endpoint \texttt{/api/users}. Una risposta HTTP 409 viene trattata come
caso non eccezionale, in quanto indica che l'utente è già registrato nel sistema.


\subsubsubsection{Membership}

\begin{figure}[H]
	\centering
	\includegraphics[width=1\textwidth]{../Assets/specifica_tecnica/Feature-membership.jpg}
	\caption{UML feature Membership}
	\label{fig:feature Membership}
\end{figure}

La feature \textit{Membership} consente all'utente autenticato di visualizzare e gestire gli inviti
ai workspace ricevuti da altri utenti. Dalla pagina dedicata, l'utente può accettare o rifiutare
ciascun invito pendente. La lista degli inviti viene recuperata dal backend al caricamento della
pagina e aggiornata automaticamente dopo ogni azione, garantendo la coerenza dei dati visualizzati.

\subsubsubsubsection{Tipi e DTO}

\textbf{Invite} è il tipo che modella un invito ricevuto dall'utente. Espone i seguenti campi:
\begin{itemize}
  \item \texttt{id: string} — identificativo univoco dell'invito;
  \item \texttt{workspaceName: string} — nome del workspace al quale l'utente è stato invitato;
  \item \texttt{senderUsername: string} — username dell'utente che ha inviato l'invito;
  \item \texttt{recipientUsername: string} — username del destinatario dell'invito;
  \item \texttt{recipientRole: string} — ruolo assegnato al destinatario nel workspace;
  \item \texttt{status: string} — stato corrente dell'invito; può assumere i valori
        \texttt{PENDING}, \texttt{ACCEPTED} o \texttt{REJECTED}.
\end{itemize}

\textbf{InviteAction} è il tipo union che rappresenta le azioni che l'utente può compiere su un
invito. Ammette i valori \texttt{'Accept'} e \texttt{'Reject'}.

\textbf{InviteState} è il tipo che raccoglie lo stato condiviso della pagina degli inviti. Espone
i seguenti campi:
\begin{itemize}
  \item \texttt{invites: Invite[]} — lista degli inviti pendenti dell'utente;
  \item \texttt{isLoading: boolean} — indica se il caricamento iniziale degli inviti è in corso;
  \item \texttt{isPending: boolean} — indica se è in corso una mutazione, ovvero un'operazione di
        accettazione o rifiuto di un invito.
\end{itemize}

\subsubsubsubsection{Componenti View}

\textbf{InvitePage} è il componente React che costituisce la pagina principale della feature.
Recupera lo stato e i handler dall'hook \texttt{useInvitePage} e li passa al componente
\texttt{InviteList}. Gestisce i casi di caricamento mostrando un messaggio testuale e, a
caricamento completato, delega la resa della lista al componente figlio.

\textbf{InviteList} è il componente che si occupa di renderizzare la collezione di inviti. Riceve
le seguenti props:
\begin{itemize}
  \item \texttt{invites: Invite[]} — lista degli inviti da visualizzare;
  \item \texttt{isPending: boolean} — se \texttt{true}, disabilita i pulsanti di azione su tutti
        gli item per prevenire operazioni concorrenti;
  \item \texttt{onAction: (membershipId: string, action: InviteAction) => void} — callback
        invocata quando l'utente accetta o rifiuta un invito.
\end{itemize}
In assenza di inviti, renderizza un messaggio di stato vuoto. In presenza di inviti, renderizza
un \texttt{InviteItem} per ciascuno, traducendo le callback \texttt{onAccept} e \texttt{onReject}
nell'unica callback \texttt{onAction} con il valore di \texttt{InviteAction} appropriato.

\textbf{InviteItem} è il componente che rappresenta il singolo invito nella lista. Riceve le
seguenti props:
\begin{itemize}
  \item \texttt{invite: Invite} — i dati dell'invito da visualizzare;
  \item \texttt{isPending: boolean} — se \texttt{true}, disabilita i pulsanti di azione;
  \item \texttt{onAccept: (membershipId: string) => void} — callback invocata al click sul
        pulsante \textit{Accetta};
  \item \texttt{onReject: (membershipId: string) => void} — callback invocata al click sul
        pulsante \textit{Rifiuta}.
\end{itemize}
Mostra il nome del workspace, le sue iniziali come avatar, lo username del mittente e il ruolo
assegnato al destinatario. I pulsanti di azione vengono disabilitati quando \texttt{isPending}
è \texttt{true}.

\subsubsubsubsection{Hook}

\textbf{useInvitePage} è il custom hook che costituisce il layer ViewModel della feature.
Implementa l'interfaccia \texttt{IInvitePageViewModel} ed è responsabile del recupero degli
inviti e della gestione delle azioni dell'utente. Internamente utilizza \texttt{useQuery} per
il fetch degli inviti e \texttt{useMutation} per le operazioni di accettazione e rifiuto,
invalidando la query \texttt{['invites']} al successo di ogni mutazione per mantenere la lista
aggiornata. Espone i seguenti membri:
\begin{itemize}
  \item \texttt{useInvitePage(): IInvitePageViewModel} — funzione costruttrice dell'hook, non
        richiede parametri in ingresso.
\end{itemize}

\textbf{IInvitePageViewModel} è l'interfaccia che definisce il contratto esposto dall'hook verso
il layer View. Estende \texttt{InviteState} aggiungendo il seguente membro:
\begin{itemize}
  \item \texttt{handleAction(membershipId: string, action: InviteAction): void} — metodo invocato
        dalla View per avviare la mutazione di accettazione o rifiuto dell'invito identificato da
        \texttt{membershipId}.
\end{itemize}

\subsubsubsubsection{Repository}

\textbf{IGetInvitesRepository} è l'interfaccia che definisce il contratto per il recupero degli
inviti dal backend. Dichiara il metodo:
\begin{itemize}
  \item \texttt{getInvites(): Promise<Invite[]>} — recupera la lista degli inviti pendenti
        dell'utente autenticato.
\end{itemize}

\textbf{GetInvitesRepository} è la classe concreta che implementa \texttt{IGetInvitesRepository}.
Utilizza la funzione \texttt{apiGet} del client HTTP condiviso per eseguire una richiesta
\texttt{GET} all'endpoint \texttt{/api/invitations}. Espone il metodo:
\begin{itemize}
  \item \texttt{getInvites(): Promise<Invite[]>} — effettua la chiamata HTTP e restituisce la
        lista degli inviti deserializzata.
\end{itemize}

\textbf{IManageInviteRepository} è l'interfaccia che definisce il contratto per la gestione di
un invito. Dichiara il metodo:
\begin{itemize}
  \item \texttt{manageInvite(membershipId: string, action: InviteAction): Promise<void>} —
        invia al backend l'azione scelta dall'utente per l'invito identificato da
        \texttt{membershipId}.
\end{itemize}

\textbf{ManageInviteRepository} è la classe concreta che implementa
\texttt{IManageInviteRepository}. Utilizza la funzione \texttt{apiPatch} del client HTTP condiviso
per eseguire una richiesta \texttt{PATCH} all'endpoint \texttt{/api/invitations/:membershipId},
passando come corpo un oggetto contenente il campo \texttt{action}. Espone il metodo:
\begin{itemize}
  \item \texttt{manageInvite(membershipId: string, action: InviteAction): Promise<void>} —
        effettua la chiamata HTTP per aggiornare lo stato dell'invito.
\end{itemize}

\subsubsubsection{CreateWorkspace}

\begin{figure}[H]
    \centering
    \includegraphics[width=1\textwidth]{../Assets/specifica_tecnica/Feature-createWorkspace.jpg}
    \caption{UML feature create Workspace}
    \label{fig:feature create Workspace}
\end{figure}


La feature \textit{CreateWorkspace} consente all'utente autenticato di creare un nuovo workspace tramite
un dialog modale. L'utente inserisce il nome del workspace, il quale viene validato lato client prima
di essere inviato al backend. In caso di errore restituito dal server, questo viene mostrato
direttamente nel form. Al successo della creazione, l'utente viene reindirizzato automaticamente
alla pagina dei repository del workspace appena creato.

\subsubsubsubsection{Tipi e DTO}

\textbf{CreateWorkspaceRequest} è il tipo che rappresenta il corpo della richiesta di creazione
workspace inviata al backend. Espone il campo \texttt{name: string}, contenente il nome scelto
dall'utente. Le informazioni sull'owner vengono ricavate autonomamente dal backend a partire dal
token JWT presente nella richiesta, senza che il client debba fornirle esplicitamente.

\textbf{WorkspaceResponse} è il tipo che modella la risposta del backend a seguito della creazione.
Espone i seguenti campi:
\begin{itemize}
  \item \texttt{id: string} — identificativo univoco del workspace appena creato;
  \item \texttt{name: string} — nome del workspace;
  \item \texttt{createdBy: string} — username dell'utente che ha creato il workspace.
\end{itemize}

\subsubsubsubsection{Componenti View}

\textbf{NewWorkspaceDialog} è il componente React che realizza il layer di presentazione della
feature. Renderizza un pulsante \textit{+ New Workspace} che, al click, apre un dialog modale
contenente un form per l'inserimento del nome del workspace. Il componente delega interamente la
logica di stato e di submission all'hook \texttt{useNewWorkspaceForm}, dal quale riceve l'istanza
\texttt{form} e l'eventuale \texttt{serverError}. Il campo nome è soggetto a validazione: l'errore
di validazione viene mostrato solo dopo che il campo è stato toccato (\textit{touched}). In caso di
errore restituito dal server, questo viene visualizzato sotto il campo. I pulsanti \textit{Cancel}
e \textit{Create Workspace} consentono rispettivamente di annullare o confermare l'operazione.

\subsubsubsubsection{Hook}

\textbf{useNewWorkspaceForm} è il custom hook che costituisce il layer ViewModel della feature.
Implementa l'interfaccia \texttt{INewWorkspaceFormViewModel} ed è responsabile della gestione dello
stato del form e della logica di submission. Internamente utilizza \texttt{useForm} di
TanStack Form con validazione tramite schema Zod (\texttt{newWorkspaceSchema}), che impone che il
nome del workspace abbia una lunghezza compresa tra 2 e 30 caratteri. Espone i seguenti membri:
\begin{itemize}
  \item \texttt{serverError: string} — messaggio di errore restituito dal backend in caso di
        fallimento della creazione; è \texttt{null} in assenza di errori;
  \item \texttt{useNewWorkspaceForm(repo: ICreateWorkspaceRepository): INewWorkspaceFormViewModel}
        — funzione costruttrice dell'hook; accetta opzionalmente un'implementazione del repository
        per facilitare il testing tramite dependency injection; al successo della creazione
        invalida la cache TanStack Query e naviga alla pagina dei repository del nuovo workspace.
\end{itemize}

\textbf{INewWorkspaceFormViewModel} è l'interfaccia che definisce il contratto esposto dall'hook
verso il layer View. Dichiara i seguenti membri:
\begin{itemize}
  \item \texttt{form: FormApi<\{name: string\}>} — istanza del form gestita da TanStack Form,
        contenente stato, validazione e handler degli eventi;
  \item \texttt{serverError: string} — eventuale messaggio di errore proveniente dal backend.
\end{itemize}

\subsubsubsubsection{Repository}

\textbf{ICreateWorkspaceRepository} è l'interfaccia che definisce il contratto del layer di accesso
ai dati per la creazione del workspace. Dichiara il metodo:
\begin{itemize}
  \item \texttt{createWorkspace(data: CreateWorkspaceRequest): Promise<WorkspaceResponse>} —
        invia la richiesta di creazione workspace al backend e restituisce i dati del workspace
        creato.
\end{itemize}

\textbf{CreateWorkspaceRepository} è la classe concreta che implementa
\texttt{ICreateWorkspaceRepository}. Utilizza la funzione \texttt{apiPost} del client HTTP
condiviso per eseguire una richiesta \texttt{POST} all'endpoint \texttt{/api/workspaces/},
passando come corpo il DTO \texttt{CreateWorkspaceRequest}. Espone il metodo:
\begin{itemize}
  \item \texttt{createWorkspace(data: CreateWorkspaceRequest): Promise<WorkspaceResponse>} —
        effettua la chiamata HTTP e restituisce la risposta deserializzata come
        \texttt{WorkspaceResponse}.
\end{itemize}


\subsubsubsection{WorkspaceMembers}

\includegraphics[width=1\textwidth]{../Assets/specifica_tecnica/Feature-WorkspaceMember.png}

La feature \textit{WorkspaceMembers} consente all'utente autenticato di gestire i membri di un
workspace. Dalla pagina dedicata è possibile visualizzare la lista dei membri correnti, invitare
nuovi utenti specificandone username e ruolo, e rimuovere membri esistenti. Tutte le operazioni
di modifica invalidano automaticamente la cache della lista membri per garantire la coerenza dei
dati visualizzati.

\subsubsubsubsection{Tipi e DTO}

\textbf{WorkspaceMember} è il tipo che modella un membro del workspace. Espone i seguenti campi:
\begin{itemize}
  \item \texttt{userId: string} — identificativo univoco dell'utente;
  \item \texttt{username: string} — nome utente visualizzato;
  \item \texttt{role: string} — ruolo del membro all'interno del workspace.
\end{itemize}

\textbf{WorkspaceRole} è il tipo union che rappresenta i ruoli assegnabili a un membro del
workspace. Ammette i valori \texttt{'project manager'}, \texttt{'tech lead'} e
\texttt{'developer'}.

\subsubsubsubsection{Componenti View}

\textbf{MembersPage} è il componente che costituisce la pagina principale della feature. Riceve
la prop \texttt{workspaceId: string} e recupera tutto lo stato necessario dall'hook aggregatore
\texttt{useMembersPage}. Compone al suo interno i componenti \texttt{InviteForm} e
\texttt{MemberList}, gestendo il caso di caricamento della lista con un messaggio testuale.

\textbf{InviteForm} è il componente che renderizza il form di invito di un nuovo membro. Riceve
le seguenti props:
\begin{itemize}
  \item \texttt{username: string} — valore corrente del campo username;
  \item \texttt{role: WorkspaceRole} — ruolo selezionato per il nuovo membro;
  \item \texttt{isPending: boolean} — se \texttt{true}, disabilita il pulsante di invio durante
        la mutazione;
  \item \texttt{isSuccess: boolean} — se \texttt{true}, mostra un messaggio di conferma invio;
  \item \texttt{error: string} — eventuale messaggio di errore restituito dal backend;
  \item \texttt{onUsernameChange: (v: string) => void} — callback invocata al cambio del campo
        username;
  \item \texttt{onRoleChange: (v: WorkspaceRole) => void} — callback invocata al cambio del
        ruolo selezionato;
  \item \texttt{onSubmit: (e: React.FormEvent) => void} — callback invocata alla submission del
        form.
\end{itemize}
Il pulsante di invio è disabilitato anche quando il campo username è vuoto.

\textbf{MemberList} è il componente che renderizza la lista dei membri del workspace. Riceve
le seguenti props:
\begin{itemize}
  \item \texttt{members: WorkspaceMember[]} — lista dei membri da visualizzare;
  \item \texttt{isRemoving: boolean} — se \texttt{true}, disabilita i pulsanti di rimozione su
        tutti gli item per prevenire operazioni concorrenti;
  \item \texttt{onRemove: (userId: string) => void} — callback invocata quando l'utente richiede
        la rimozione di un membro.
\end{itemize}
In assenza di membri mostra un messaggio di stato vuoto. In presenza di membri renderizza un
\texttt{MemberItem} per ciascuno.

\textbf{MemberItem} è il componente che rappresenta il singolo membro nella lista. Riceve le
seguenti props:
\begin{itemize}
  \item \texttt{member: WorkspaceMember} — i dati del membro da visualizzare;
  \item \texttt{isRemoving: boolean} — se \texttt{true}, disabilita il pulsante di rimozione;
  \item \texttt{onRemove: (userId: string) => void} — callback invocata al click sul pulsante
        \textit{Rimuovi}.
\end{itemize}
Mostra le iniziali dello username come avatar, il nome utente e il ruolo del membro.

\subsubsubsubsection{Hook}

\textbf{useMembersPage} è il custom hook aggregatore che costituisce il layer ViewModel della
feature. Implementa l'interfaccia \texttt{IMembersPageViewModel} componendo internamente i tre
hook specializzati \texttt{useGetMembers}, \texttt{useRemoveMember} e \texttt{useInviteMember}.
Accetta il parametro \texttt{workspaceId: string} ed espone i seguenti membri:
\begin{itemize}
  \item \texttt{members: WorkspaceMember[]} — lista dei membri correnti del workspace;
  \item \texttt{membersLoading: boolean} — indica se il caricamento della lista è in corso;
  \item \texttt{isRemoving: boolean} — indica se una rimozione è in corso;
  \item \texttt{removeMember(userId: string): void} — avvia la mutazione di rimozione del membro
        identificato da \texttt{userId};
  \item \texttt{inviteForm: IInviteMemberViewModel} — oggetto che espone lo stato e i handler
        del form di invito.
\end{itemize}

\textbf{useGetMembers} è il custom hook responsabile del recupero della lista dei membri.
Utilizza \texttt{useQuery} di TanStack Query con query key \texttt{['members', workspaceId]},
abilitata solo se \texttt{workspaceId} è definito. Accetta il parametro
\texttt{workspaceId: string} e restituisce i dati della query inclusi \texttt{data} e
\texttt{isLoading}.

\textbf{useInviteMember} è il custom hook responsabile della logica di invito di un nuovo membro.
Implementa l'interfaccia \texttt{IInviteMemberViewModel} e utilizza \texttt{useMutation} per
invocare \texttt{InviteMemberRepository.inviteMember}. Al successo della mutazione invalida la
query \texttt{['members', workspaceId]} e reimposta i campi del form. Espone i seguenti membri:
\begin{itemize}
  \item \texttt{username: string} — valore corrente del campo username;
  \item \texttt{setUsername(username: string): void} — aggiorna il valore del campo username;
  \item \texttt{role: WorkspaceRole} — ruolo correntemente selezionato;
  \item \texttt{setRole(role: WorkspaceRole): void} — aggiorna il ruolo selezionato;
  \item \texttt{isPending: boolean} — indica se la mutazione di invito è in corso;
  \item \texttt{isSuccess: boolean} — indica se l'invito è stato inviato con successo;
  \item \texttt{error: Error} — eventuale errore restituito dal backend;
  \item \texttt{handleSubmit(e: SubmitEvent): void} — gestisce la submission del form,
        prevenendo l'invio se lo username è vuoto.
\end{itemize}

\textbf{useRemoveMember} è il custom hook responsabile della rimozione di un membro. Utilizza
\texttt{useMutation} per invocare \texttt{RemoveMemberRepository.removeMember} e al successo
invalida la query \texttt{['members', workspaceId]}. Accetta il parametro
\texttt{workspaceId: string} e restituisce direttamente il risultato di \texttt{useMutation},
esponendo tra gli altri \texttt{mutate} e \texttt{isPending}.

\textbf{IInviteMemberViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useInviteMember} verso il layer View. Dichiara i membri \texttt{username},
\texttt{setUsername}, \texttt{role}, \texttt{setRole}, \texttt{isPending}, \texttt{isSuccess},
\texttt{error} e \texttt{handleSubmit}, con le stesse semantiche descritte per l'hook.

\textbf{IMembersPageViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useMembersPage} verso il layer View. Dichiara i membri \texttt{members},
\texttt{membersLoading}, \texttt{isRemoving}, \texttt{removeMember} e \texttt{inviteForm},
con le stesse semantiche descritte per l'hook.

\subsubsubsubsection{Repository}

\textbf{IGetMembersRepository} è l'interfaccia che definisce il contratto per il recupero dei
membri. Dichiara il metodo:
\begin{itemize}
  \item \texttt{getMembers(workspaceId: string): Promise<WorkspaceMember[]>} — recupera la lista
        dei membri del workspace identificato da \texttt{workspaceId}.
\end{itemize}

\textbf{GetMembersRepository} è la classe concreta che implementa \texttt{IGetMembersRepository}.
Utilizza \texttt{apiGet} per eseguire una richiesta \texttt{GET} all'endpoint
\texttt{/api/workspaces/:workspaceId/users}. Espone il metodo:
\begin{itemize}
  \item \texttt{getMembers(workspaceId: string): Promise<WorkspaceMember[]>} — effettua la
        chiamata HTTP e restituisce la lista dei membri deserializzata.
\end{itemize}

\textbf{IInviteMemberRepository} è l'interfaccia che definisce il contratto per l'invito di un
nuovo membro. Dichiara il metodo:
\begin{itemize}
  \item \texttt{inviteMember(workspaceId: string, recipientUsername: string, recipientRole:
        WorkspaceRole): Promise<void>} — invia al backend la richiesta di invito per l'utente
        specificato con il ruolo indicato.
\end{itemize}

\textbf{InviteMemberRepository} è la classe concreta che implementa
\texttt{IInviteMemberRepository}. Utilizza \texttt{apiPost} per eseguire una richiesta
\texttt{POST} all'endpoint \texttt{/api/invitations}, passando come corpo un oggetto contenente
\texttt{workspaceId}, \texttt{recipientUsername} e \texttt{recipientRole}. Espone il metodo:
\begin{itemize}
  \item \texttt{inviteMember(workspaceId: string, recipientUsername: string, recipientRole:

\subsubsubsection{Scan}

\begin{figure}[H]

La feature \textit{Scan} consente all'utente di avviare e interrompere una scansione su un
repository GitHub per un determinato branch. Il componente dedicato mostra un pulsante di avvio
che, una volta premuto, dà inizio alla scansione e si trasforma in un pulsante di interruzione.
Lo stato della scansione viene monitorato tramite polling periodico al backend. Al raggiungimento
di uno stato terminale, la cache del report viene invalidata automaticamente per aggiornare i
risultati visualizzati. È presente inoltre un meccanismo di fallback che, trascorso un timeout,
interroga direttamente il report per verificare il completamento della scansione in caso di
mancata ricezione dello stato terminale.

\subsubsubsubsection{Tipi e DTO}

\textbf{StartScanInfo} è il tipo che raccoglie le informazioni necessarie per avviare una
scansione. Espone i seguenti campi:
\begin{itemize}
  \item \texttt{workspaceId: string} — identificativo del workspace a cui appartiene il repository;
  \item \texttt{repositoryId: string} — identificativo del repository da scansionare;
  \item \texttt{branch: string} — nome del branch su cui eseguire la scansione.
\end{itemize}

\subsubsubsubsection{Componenti View}

\textbf{ScanButton} è il componente React che costituisce l'unico elemento di presentazione della
feature. Viene utilizzato all'interno della \texttt{ReportPage} e coordina internamente i tre
hook della feature. Riceve le seguenti props:
\begin{itemize}
  \item \texttt{workspaceId: string} — identificativo del workspace;
  \item \texttt{repositoryId: string} — identificativo del repository;
  \item \texttt{branch: string} — branch selezionato; se vuoto, il pulsante di avvio viene
        disabilitato;
  \item \texttt{onCompleted: () => void} — callback opzionale invocata al completamento con
        successo della scansione, utilizzata dalla \texttt{ReportPage} per invalidare la cache
        del report;
  \item \texttt{onScanActiveChange: (active: boolean) => void} — callback opzionale invocata
        ogni volta che lo stato attivo della scansione cambia, utilizzata per sopprimere lo
        skeleton di caricamento del report durante la scansione.
\end{itemize}
Il componente distingue due stati visivi principali: quando nessuna scansione è attiva mostra
il pulsante \textit{Lancia scansione}; quando una scansione è in corso mostra il pulsante
\textit{Ferma scansione}. Gli eventuali errori di avvio o di interruzione vengono mostrati
sotto il pulsante. Internamente gestisce un meccanismo di fallback basato su \texttt{setTimeout}
e polling del report tramite \texttt{useQueryClient} per garantire il completamento anche in
assenza di aggiornamenti di stato dal backend entro 90 secondi.

\subsubsubsubsection{Hook}

\textbf{useScan} è il custom hook responsabile dell'avvio della scansione. Implementa
l'interfaccia \texttt{IScanViewModel} e utilizza internamente \texttt{useMutation} di TanStack
Query per invocare \texttt{ScanRepository.requestScan}. Accetta in ingresso un payload di tipo
\texttt{StartScanInfo} e restituisce i seguenti membri:
\begin{itemize}
  \item \texttt{triggerScan(): void} — avvia la mutazione di richiesta scansione;
  \item \texttt{scanId: string} — identificativo della scansione restituito dal backend al
        successo della mutazione; è \texttt{undefined} prima del successo;
  \item \texttt{isPending: boolean} — indica se la richiesta di avvio è in corso;
  \item \texttt{isSuccess: boolean} — indica se la scansione è stata avviata con successo;
  \item \texttt{error: Error} — eventuale errore restituito dal backend;
  \item \texttt{reset(): void} — reimposta lo stato della mutazione al valore iniziale.
\end{itemize}

\textbf{useScanStatus} è il custom hook responsabile del monitoraggio dello stato di una
scansione in corso. Implementa l'interfaccia \texttt{IScanStatusViewModel} e utilizza
internamente \texttt{useQuery} con refetch automatico ogni 3 secondi, che si interrompe
automaticamente al raggiungimento di uno stato terminale (\texttt{completed}, \texttt{stopped},
\texttt{error}). Accetta in ingresso il \texttt{scanId} e la query è abilitata solo se questo
è definito. Espone i seguenti membri:
\begin{itemize}
  \item \texttt{scanStatus: string} — stato corrente della scansione restituito dal backend;
  \item \texttt{isStatusLoading: boolean} — indica se il primo caricamento dello stato è in corso.
\end{itemize}

\textbf{useStopScan} è il custom hook responsabile dell'interruzione di una scansione in corso.
Implementa l'interfaccia \texttt{IStopScanViewModel} e utilizza internamente \texttt{useMutation}
per invocare \texttt{ScanRepository.stopScan}. Accetta opzionalmente una callback
\texttt{onSettled} invocata al termine della mutazione, indipendentemente dal suo esito, utilizzata
dal \texttt{ScanButton} per rimuovere la query di stato e resettare la mutazione di avvio. Espone
i seguenti membri:
\begin{itemize}
  \item \texttt{triggerStop(scanId: string): void} — avvia la mutazione di interruzione della
        scansione identificata da \texttt{scanId};
  \item \texttt{isStopping: boolean} — indica se la richiesta di interruzione è in corso;
  \item \texttt{stopError: Error} — eventuale errore restituito dal backend durante
        l'interruzione;
  \item \texttt{resetStop(): void} — reimposta lo stato della mutazione al valore iniziale.
\end{itemize}

\textbf{IScanViewModel} è l'interfaccia che definisce il contratto esposto da \texttt{useScan}
verso il layer View. Dichiara i membri \texttt{triggerScan}, \texttt{scanId}, \texttt{isPending},
\texttt{isSuccess}, \texttt{error} e \texttt{reset}, con le stesse semantiche descritte per
l'hook.

\textbf{IScanStatusViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useScanStatus}. Dichiara i membri \texttt{scanStatus} e \texttt{isStatusLoading}.

\textbf{IStopScanViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useStopScan}. Dichiara i membri \texttt{triggerStop}, \texttt{isStopping},
\texttt{stopError} e \texttt{resetStop}.

\subsubsubsubsection{Repository}

\textbf{IScanRepository} è l'interfaccia che definisce il contratto del layer di accesso ai dati
per la feature. Dichiara i seguenti metodi:
\begin{itemize}
  \item \texttt{requestScan(payload: StartScanInfo): Promise<string>} — invia la richiesta di
        avvio scansione al backend e restituisce l'identificativo della scansione creata;
  \item \texttt{stopScan(scanId: string): Promise<void>} — invia la richiesta di interruzione
        della scansione identificata da \texttt{scanId};
  \item \texttt{getScanStatus(scanId: string): Promise<string>} — recupera lo stato corrente
        della scansione identificata da \texttt{scanId}.
\end{itemize}

\textbf{ScanRepository} è la classe concreta che implementa \texttt{IScanRepository}. Utilizza
le funzioni \texttt{apiPost}, \texttt{apiPatch} e \texttt{apiGet} del client HTTP condiviso.
Espone i seguenti metodi:
\begin{itemize}
  \item \texttt{requestScan(payload: StartScanInfo): Promise<string>} — esegue una richiesta
        \texttt{POST} all'endpoint \texttt{/api/scan} con il payload fornito e restituisce il
        campo \texttt{scanId} dalla risposta;
  \item \texttt{stopScan(scanId: string): Promise<void>} — esegue una richiesta \texttt{PATCH}
        all'endpoint \texttt{/api/scan/:scanId} per interrompere la scansione;
  \item \texttt{getScanStatus(scanId: string): Promise<string>} — esegue una richiesta
        \texttt{GET} all'endpoint \texttt{/api/scan/:scanId/status} e restituisce lo stato
        corrente come stringa.
\end{itemize}

  \item \texttt{metadata?: Metadata} — metadati della scansione, opzionali.
\end{itemize}

\textbf{Summary} è il tipo che modella il riepilogo globale del report. Espone i seguenti campi:
\begin{itemize}
  \item \texttt{summary: string} — testo descrittivo del riepilogo in formato Markdown;
  \item \texttt{mark: number} — voto globale del report su scala da 0 a 10.
\end{itemize}

\textbf{Metadata} è il tipo che raccoglie i metadati della scansione. Espone i seguenti campi:
\begin{itemize}
  \item \texttt{target: Target} — informazioni sul repository scansionato;
  \item \texttt{startScanTime: string} — timestamp di inizio scansione in formato ISO 8601;
  \item \texttt{endScanTime: string} — timestamp di fine scansione in formato ISO 8601.
\end{itemize}

\textbf{Target} è il tipo che identifica il repository scansionato. Espone i campi
\texttt{owner: string}, \texttt{repository: string} e \texttt{branch: string}.

\textbf{ReportData} è il tipo che raccoglie i risultati delle analisi suddivisi per categoria.
Espone i seguenti campi:
\begin{itemize}
  \item \texttt{testReport: TestReport} — risultati dell'analisi dei test;
  \item \texttt{vulnerabilitiesReport: VulnerabilitiesReport} — risultati dell'analisi delle

  \item \texttt{depsReport: DepsReport} — risultati dell'analisi delle dipendenze vulnerabili;
  \item \texttt{docsReport: DocsReport} — risultati dell'analisi della documentazione;
  \item \texttt{techReport: TechReport} — risultati dell'analisi tecnologica.

\textbf{TestReport} è il tipo che modella i risultati dell'analisi dei test. Espone i seguenti
campi:
\begin{itemize}

  \item \texttt{failedTests: FailedTest[]} — lista dei test falliti;
  \item \texttt{testsRun: number} — numero totale di test eseguiti.
\end{itemize}

\textbf{CoverageReport} è il tipo che modella le percentuali di coverage. Espone i campi
\texttt{statements: number}, \texttt{branches: number}, \texttt{functions: number} e
\texttt{lines: number}.

\textbf{FailedTest} è il tipo che modella un test fallito. Espone i campi \texttt{name: string},
\texttt{path: string} e \texttt{messageSummary: string}.

\textbf{VulnerabilitiesReport} è il tipo che modella i risultati dell'analisi delle vulnerabilità
nel codice sorgente. Espone i seguenti campi:
\begin{itemize}

  \item \texttt{vulnCounts?: VulnCounts} — conteggio delle vulnerabilità per livello di severità,

  \item \texttt{mark: number} — voto della sicurezza del codice su scala da 0 a 10.
\end{itemize}

\textbf{CodeVulnerability} è il tipo che modella una singola vulnerabilità nel codice. Espone
i campi \texttt{id: string}, \texttt{path: string}, \texttt{description: string},
\texttt{severity: number}, \texttt{impact: string}, \texttt{category: string},

\textbf{DepsReport} è il tipo che modella i risultati dell'analisi delle dipendenze vulnerabili.

\subsubsubsubsection{Hook}

\textbf{useReportPage} è il custom hook aggregatore che costituisce il layer ViewModel della
feature. Implementa l'interfaccia \texttt{IReportPageViewModel} componendo internamente
\texttt{useGetBranches} e \texttt{useGetReport}. Gestisce la selezione automatica del branch
tramite \texttt{useEffect}: al caricamento dei branch seleziona \texttt{develop} se presente,
altrimenti il primo disponibile; la selezione automatica non sovrascrive una selezione
effettuata manualmente dall'utente, tracciata tramite un flag interno \texttt{userSelected}.
Accetta il parametro \texttt{repositoryId: string} ed espone i seguenti membri tramite
\texttt{IReportPageViewModel}:
\begin{itemize}
  \item \texttt{branches: string[]} — lista dei branch disponibili per il repository;
  \item \texttt{branchesLoading: boolean} — indica se il caricamento dei branch è in corso;
  \item \texttt{selectedBranch: string} — branch attualmente selezionato;
  \item \texttt{setSelectedBranch(branch: string): void} — aggiorna il branch selezionato
        marcando la selezione come manuale;
  \item \texttt{report: ReportInfo} — dati del report per il branch selezionato;
  \item \texttt{reportLoading: boolean} — indica se il caricamento del report è in corso;
  \item \texttt{reportError: Error} — eventuale errore nel caricamento del report.
\end{itemize}

\textbf{useGetBranches} è il custom hook responsabile del recupero dei branch disponibili per
un repository. Utilizza \texttt{useQuery} con query key \texttt{['branches', repositoryId]},
abilitata solo se \texttt{repositoryId} è definito. Accetta il parametro
\texttt{repositoryId: string} e restituisce i dati della query.

\textbf{useGetReport} è il custom hook responsabile del recupero del report per un branch
specifico. Utilizza \texttt{useQuery} con query key \texttt{['report', repositoryId, branch]},
abilitata solo se \texttt{branch} è definito, e con \texttt{retry: false} per non ripetere
la richiesta in caso di errore. Accetta i parametri \texttt{repositoryId: string} e
\texttt{branch: string | undefined} e restituisce i dati della query.

\textbf{useGetRepository} è il custom hook responsabile del recupero dei metadati del
repository visualizzato. Utilizza \texttt{useQuery} con query key
\texttt{['repository', repositoryId]}, abilitata solo se \texttt{repositoryId} è definito.
Accetta il parametro \texttt{repositoryId: string} e restituisce i dati del repository
corrispondente.

\textbf{useUpdateToken} è il custom hook responsabile dell'aggiornamento del token di accesso
di un repository privato. Utilizza \texttt{useMutation} per invocare
\texttt{UpdateTokenRepository.updateToken}. Prima di inviare la richiesta esegue una
validazione lato client del token tramite espressione regolare, accettando token nel formato
\texttt{ghp\_*} o \texttt{github\_pat\_*}. Accetta i parametri \texttt{workspaceId: string},
\texttt{repositoryId: string} e la callback opzionale \texttt{onSaved} invocata al successo
della mutazione. Espone i seguenti membri:
\begin{itemize}
  \item \texttt{token: string} — valore corrente del campo token;
  \item \texttt{setToken(token: string): void} — aggiorna il valore del campo token;
  \item \texttt{isPending: boolean} — indica se la mutazione è in corso;
  \item \texttt{clientError: string} — eventuale messaggio di errore di validazione lato client;
  \item \texttt{serverError: Error} — eventuale errore restituito dal backend;
  \item \texttt{handleSubmit(e: React.SyntheticEvent<HTMLFormElement>): void} — gestisce la
        submission eseguendo prima la validazione del formato del token.
\end{itemize}

\textbf{IReportPageViewModel} è l'interfaccia che definisce il contratto esposto da
\texttt{useReportPage} verso il layer View. Dichiara i membri \texttt{branches},
\texttt{branchesLoading}, \texttt{selectedBranch}, \texttt{setSelectedBranch}, \texttt{report},
\texttt{reportLoading} e \texttt{reportError}, con le stesse semantiche descritte per l'hook.

\subsection{Backend}
\subsubsection{Introduzione}
Il backend sfrutta l'approccio modulare offerto dal framework NestJS e adotta un'architettura layered, facendo in modo che i vari moduli siano organizzati in strati logici distinti, per garantire una chiara separazione delle responsabilità.
\\ \\
L'architettura layered si articola sui seguenti strati:
\begin{itemize}
  \item Presentation Layer: funge da punto di ingresso per le richieste HTTP provenienti dal frontend, si occupa esclusivamente della ricezione degli input e della restituzione delle risposte.
  \item Business Logic Layer: in questo livello viene implementata la logica di business.
  \item Persistence Layer: gestisce l'interazione con il database, questo strato isola le operazioni di persistenza dal resto dell'applicazione.
\end{itemize}

\subsubsection{Design pattern}
\subsubsubsection{Dependency injection}
In NestJS, la Dependency Injection è il meccanismo di default per risolvere le dipendenze tra classi. Attraverso il decoratore @Injectable(), i servizi vengono forniti a controller o altri provider in modo automatico. Tale architettura minimizza la dipendenza tra classi e ne semplifica il testing.
\subsubsubsection{Repository pattern}
Agisce da ponte per separare le regole di business dai dettagli di persistenza. Centralizzando le operazioni verso il database. In questo modo viene nascosta la complessità delle query ai service.

Il modulo \texttt{WorkspaceRepository} è responsabile della gestione delle associazioni logiche tra i workspace e le repository esterne. Garantisce inolte che ogni operazione passi attraverso le corrette verifiche di sicurezza e delega le interazioni con i provider esterni al modulo Repository.
\subsubsubsubsection{Tipi}
\begin{itemize}
  \item \textbf{Application logic}

      A differenza dei moduli core, il modulo \texttt{WorkspaceRepository} non definisce Business Object (BO) complessi per le operazioni di scrittura o aggiornamento, in quanto i layer di servizio scambiano direttamente tipi primitivi come identificativi (ID) e URL, delegando la costruzione degli oggetti di dominio al modulo \texttt{Repository}.
    \end{itemize}

  \item \textbf{Persistence logic}\\
  A differenza del modulo WorkspaceManager, questo modulo non definisce un oggetto di persistenza autonomo (Data) né un proprio Schema Mongoose. Le sue operazioni di persistenza si limitano all'estrazione e alla manipolazione diretta del campo relazionale repositories (un array di stringhe che funge da elenco di chiavi esterne) all'interno del preesistente WorkspaceDocument.

	Il \textbf{Container} è il componente autonomo del sistema CodeGuardian incaricato dell'esecuzione materiale dell'analisi di un repository. Viene istanziato dal backend come task isolato su \textit{AWS ECS (Fargate)} ogni volta che una nuova scansione viene avviata, e comunica con il backend tramite webhook al termine dell'elaborazione.
 
	Il Container è un'applicazione \textbf{NestJS standalone}, priva di un server HTTP in ascolto: il suo unico punto di ingresso è il metodo \texttt{AppService.run()}, invocato direttamente all'avvio del processo. Tutta la configurazione necessaria (credenziali GitHub, token Bedrock, URL di callback) viene iniettata tramite la variabile d'ambiente \texttt{REPORT\_CALLBACK\_TOKEN}, un JWT firmato generato dal backend nel momento in cui il task viene lanciato.
 
	Il flusso di esecuzione del Container segue le seguenti fasi:
	\begin{enumerate}
		\item Decodifica del \texttt{REPORT\_CALLBACK\_TOKEN} e validazione dei dati di configurazione;
		\item Validazione della raggiungibilità del repository GitHub e delle credenziali AWS Bedrock;
		\item Avvio della pipeline di analisi tramite \texttt{ScanService}, che delega l'esecuzione al grafo LangGraph orchestrato da \texttt{OrchestratorService};
		\item Invio del report prodotto all'URL di successo, oppure notifica dell'errore all'URL di fallimento, tramite \texttt{ReporterService}.
	\end{enumerate}
 
 %Da capire se si puo parlare di moduli o meno
	L'architettura interna è organizzata in tre sezioni principali:
	\begin{itemize}
		\item \textbf{AppModule}: modulo radice che aggrega gli altri moduli e configura le dipendenze globali;
		\item \textbf{ScanModule}: incapsula la logica di scansione e delega l'esecuzione del grafo di analisi all'\texttt{OrchestratorModule};
		\item \textbf{ReporterModule}: gestisce la comunicazione dei risultati verso il backend al termine dell'elaborazione.
	\end{itemize}

  \centering
  \includegraphics[width=0.9\textwidth]{../Assets/specifica_tecnica/container/moduli_container.png}
  \caption{Moduli del container di scansione}

\subsubsection{Design pattern}
All'interno del container non sono stati utilizzati pattern particolari, salvo il \textit{Dependency Injection}, di cui si parlerà nella prossima sezione.

Per questioni di praticità in fase di testing è stato scelto di utilizzare classi \textit{helper} non esportate, dunque private, per ciascun modulo, qualora necessario.

\subsubsubsection{Dependency Injection}
Il pattern \textit{Dependency Injection}, gestito direttamente da NestJS, è stato ampliamente utilizzato all'interno del container. Ciò ha permesso di diminuire il livello di dipendenza tra le classi e facilitare la fase di test.

\subsubsubsection{OrchestratorService}
\texttt{OrchestratorService} è responsabile della costruzione e dell'esecuzione del grafo

  restituisce il percorso locale al repository. Se il repository è già presente
  localmente, restituisce il percorso senza eseguire nuovamente il clone. In caso di
  errore durante il clone, viene sollevata un'eccezione di tipo \texttt{Error}.
\end{itemize}

\subsubsection{Nodo analisi della documentazione}
\begin{figure}[h!]
\centering
\includegraphics[width=0.3\textwidth]{../Assets/specifica_tecnica/container/DocsModule.png}
\caption{UML DocsModule}
\label{fig:docs_node}
\end{figure}
\texttt{DocsModule} contiene due \textit{provider}: \texttt{DocsNodeService},
registrato tramite il token \texttt{DOCS\_NODE\_SERVICE\_TOKEN}, e \texttt{DocsNodeHelper}.
Il modulo esporta il token del \texttt{DocsNodeService} per consentire l'iniezione
di interfacce implementate da tale service.
 
\subsubsubsection{DocsNodeService}
\texttt{DocsNodeService} implementa l'interfaccia \texttt{INodeScanService} ed è
responsabile della valutazione della qualità della documentazione presente nel repository
analizzato, considerando sia il file \texttt{README.md} sia i commenti nel codice sorgente.
\begin{itemize}
  \item \texttt{scan(\{ repoPath \}: \{ repoPath: string \}): Promise<Partial<WorkflowState>>}:
  richiede un percorso valido al repository locale; delega all'helper la raccolta dei file
  di testo rilevanti e l'intera analisi documentale, restituendo un aggiornamento parziale
  del \texttt{WorkflowState} contenente il \texttt{DocsReport} prodotto. Non solleva eccezioni.
\end{itemize}
 
\subsubsubsection{DocsNodeHelper}
\texttt{DocsNodeHelper} fornisce al \texttt{DocsNodeService} le operazioni necessarie alla
raccolta dei file, all'analisi tramite modello LLM e al calcolo del voto finale. Il modello
utilizzato è configurabile tramite variabili d'ambiente e opera con temperatura zero per
garantire output deterministici.
\begin{itemize}
  \item \texttt{collectTextFiles(dirPath: string): string[]}:
  richiede un percorso valido alla directory radice; restituisce la lista dei percorsi
  assoluti di tutti i file di testo analizzabili, escludendo le directory irrilevanti
  (quali \texttt{node\_modules} e \texttt{dist}) e i file di dimensione superiore a 100\,KB.
  Non solleva eccezioni.
 
  \item \texttt{analyzeRepoDocumentation(repoPath: string, files: string[]): Promise<DocsReport>}:
  richiede un percorso valido al repository e la lista dei file raccolti; avvia in parallelo
  l'analisi del README e quella dei commenti nel codice, poi ne combina i risultati invocando
  il calcolo del voto finale. Restituisce il \texttt{DocsReport} completo. Non solleva eccezioni.
 
  \item \texttt{analyzeReadme(repoPath: string): Promise<string>}:
  richiede un percorso valido al repository; legge il file \texttt{README.md} e lo sottopone
  al modello LLM, che valuta panoramica e struttura del progetto (fino a 5 punti), completezza
  delle istruzioni di build ed esempi di utilizzo (fino a 3 punti) e leggibilità del testo
  tramite l'indice di Gulpease per l'italiano o la formula di Flesch per l'inglese (fino a
  2 punti). In assenza del README, restituisce una stringa di avviso predefinita senza
  interpellare il modello. Non solleva eccezioni.
 
  \item \texttt{analyzeCodeComments(repoPath: string, files: string[]): Promise<string>}:
  richiede un percorso valido al repository e la lista dei file da analizzare; suddivide il
  contenuto in batch da 512\,KB e li invia al modello in parallelo per valutare presenza
  di commenti sulle funzioni pubbliche (fino a 2 punti), coerenza tra commenti e codice
  (fino a 4 punti) e completezza della documentazione (fino a 4 punti). Se i batch sono
  più di uno, i report parziali vengono sintetizzati in un'ulteriore chiamata al modello.
  Restituisce il report testuale in Markdown. Non solleva eccezioni.
 
  \item \texttt{extractMark(readmeReport: string, commentReport: string): Promise<number>}:
  richiede i due report testuali prodotti dalle analisi precedenti; chiede al modello di
  estrarne i punteggi numerici e di calcolare il voto finale come media delle due somme
  parziali, ciascuna su base 10. Restituisce un numero decimale con al più una cifra dopo
  la virgola. In caso di errore nella chiamata al modello, restituisce 0 senza propagare
  eccezioni.
\end{itemize}
 
\subsubsection{Nodo analisi della sicurezza del codice}
\begin{figure}[h!]
\centering
\includegraphics[width=0.3\textwidth]{../Assets/specifica_tecnica/container/SecurityModule.png}
\caption{UML SecurityModule}
\label{fig:security_module}
\end{figure}
\texttt{SecurityModule} contiene due \textit{provider}: \texttt{SecurityNodeService},
registrato tramite il token \newline \texttt{SECURITY\_NODE\_SERVICE\_TOKEN}, e \texttt{SecurityNodeHelper}.
Il modulo esporta il token del \newline \texttt{SecurityNodeService} per consentire l'iniezione
di interfacce implementate da tale service. L'output di questo nodo alimenta direttamente
il nodo \texttt{RemediationNode}.
 
\subsubsubsection{SecurityNodeService}
\texttt{SecurityNodeService} implementa l'interfaccia \texttt{INodeScanService} ed è
responsabile dell'identificazione delle vulnerabilità presenti nel codice sorgente tramite
analisi statica con \textit{Semgrep}.
\begin{itemize}
  \item \texttt{scan(\{ repoPath \}: \{ repoPath: string \}): Promise<Partial<WorkflowState>>}:
  richiede un percorso valido al repository locale; verifica la disponibilità di
  \textit{Semgrep}, esegue la scansione, traduce le descrizioni delle vulnerabilità in
  italiano tramite il modello LLM e calcola il voto di sicurezza. Restituisce un
  aggiornamento parziale del \texttt{WorkflowState} contenente il
  \texttt{VulnerabilitiesReport} e il percorso del file JSON grezzo prodotto da
  \textit{Semgrep}, necessario al nodo di remediation. In caso di errore in qualsiasi
  fase, restituisce un oggetto vuoto senza propagare eccezioni.
\end{itemize}
 
\subsubsubsection{SecurityNodeHelper}
\texttt{SecurityNodeHelper} fornisce al \texttt{SecurityNodeService} le operazioni
necessarie all'esecuzione di \textit{Semgrep}, all'interpretazione dei risultati e al
calcolo del voto finale.
\begin{itemize}
  \item \texttt{buildReportPath(repoName: string): string}:
  richiede il nome del repository; restituisce il percorso del file JSON in cui
  \textit{Semgrep} salverà i risultati, includendo un timestamp per garantire l'unicità
  del nome. Non solleva eccezioni.
 
  \item \texttt{isSemgrepInstalled(): Promise<boolean>}:
  verifica che \textit{Semgrep} sia presente e raggiungibile nel \texttt{PATH} di sistema;
  restituisce \texttt{true} se lo strumento è disponibile, \texttt{false} altrimenti.
  Non solleva eccezioni.
 
  \item \texttt{executeSemgrep(repoPath: string, reportPath: string): Promise<void>}:
  richiede un percorso valido al repository e il percorso di destinazione del report;
  esegue \textit{Semgrep} con rilevamento automatico del ruleset, escludendo le directory
  di build e le dipendenze esterne, e salva i risultati in formato JSON. Propaga
  un'eccezione di tipo \texttt{Error} al chiamante in caso di fallimento.
 
  \item \texttt{parseResults(reportPath: string): Promise<VulnerabilityUnit[]>}:
  richiede il percorso del file JSON prodotto da \textit{Semgrep}; lo legge e trasforma
  ogni risultato in un oggetto \texttt{VulnerabilityUnit} con identificatore, percorso
  del file, descrizione, severità, categoria e riferimenti agli standard CWE e OWASP.
  Il campo \texttt{remediation} viene inizializzato vuoto. Propaga un'eccezione di tipo
  \texttt{Error} in caso di file non leggibile o JSON non valido.
 
  \item \texttt{translateDescriptions(units: VulnerabilityUnit[]): Promise<VulnerabilityUnit[]>}:
  richiede la lista delle vulnerabilità rilevate; invia le descrizioni originali in inglese
  al modello LLM, che le restituisce tradotte in italiano. Restituisce la lista aggiornata
  con le traduzioni rimappate. In caso di errore nella chiamata al modello, restituisce la
  lista originale invariata senza propagare eccezioni.
 
  \item \texttt{getMark(vulnerabilities: VulnerabilityUnit[]): number}:
  richiede la lista delle vulnerabilità; calcola il voto come media delle severity, dove
  \texttt{INFO} vale 10, \texttt{WARNING} vale 5 e \texttt{ERROR} vale 0. Restituisce 10
  in assenza di vulnerabilità. Non solleva eccezioni.
\end{itemize}

 
\subsubsection{Nodo creazione delle remediation}
 \begin{figure}[h!]
\centering
\includegraphics[width=0.4\textwidth]{../Assets/specifica_tecnica/container/RemediationNodeService.png}
\caption{UML RemediationModule}
\label{fig:remediation_module}
\end{figure}
\texttt{RemediationModule} contiene due \textit{provider}: \texttt{RemediationNodeService},
registrato tramite il token \texttt{REMEDIATION\_NODE\_SERVICE\_TOKEN}, e
\texttt{RemediationNodeHelper}. Il modulo esporta il token del
\texttt{RemediationNodeService} per consentire l'iniezione di interfacce implementate
da tale service. Questo nodo è il successore diretto di \texttt{SecurityNode} nel grafo
e si occupa di arricchire le vulnerabilità già rilevate con le relative azioni correttive.
 
\subsubsubsection{RemediationNodeService}
\texttt{RemediationNodeService} implementa l'interfaccia \texttt{INodeScanService} ed è
responsabile della generazione delle remediation per le vulnerabilità identificate da
\texttt{SecurityNode}. A differenza degli altri nodi, il metodo \texttt{scan} riceve
in input gli output del nodo precedente anziché il percorso della repository.
\begin{itemize}
  \item \texttt{scan(\{ vulnerabilitiesReport, vulnerabilitiesReportPath \}):}
  \newline \texttt{Promise<Partial<WorkflowState>>}: \newline
  richiede il report delle vulnerabilità già tradotte e il percorso del file grezzo
  prodotto da \textit{Semgrep}; verifica la presenza e l'accessibilità del file, poi
  delega all'helper il raggruppamento dei risultati per file sorgente e l'elaborazione
  in parallelo di ciascun gruppo. Per ogni file, applica le remediation generate sulle
  corrispondenti \texttt{VulnerabilityUnit}. Restituisce il \texttt{VulnerabilitiesReport}
  aggiornato con le remediation compilate. In assenza di vulnerabilità o in caso di errore
  nell'accesso al file, restituisce il report originale invariato o un oggetto vuoto
  senza propagare eccezioni. Gli errori relativi a singoli file vengono gestiti
  individualmente, senza interrompere l'elaborazione degli altri.
\end{itemize}
 
\subsubsubsection{RemediationNodeHelper}
\texttt{RemediationNodeHelper} fornisce al \texttt{RemediationNodeService} le operazioni
necessarie al raggruppamento delle vulnerabilità, alla lettura dei file sorgente, alla
generazione delle remediation tramite modello LLM e alla loro applicazione sul report.
Il limite di token per risposta è impostato a 15\,000, valore superiore rispetto agli
altri nodi, per accomodare la generazione di remediation articolate su file con molte
vulnerabilità.
\begin{itemize}
 
  \item \texttt{groupResultsByFile(results: SemgrepResult[]): Map<string, SemgrepResult[]>}:
  richiede la lista dei risultati grezzi prodotti da \textit{Semgrep}; li raggruppa per
  percorso del file affetto e restituisce una mappa che associa ciascun percorso alla
  lista delle vulnerabilità rilevate in quel file, garantendo che ogni file venga
  elaborato in un'unica chiamata al modello. Non solleva eccezioni.
 
  \item \texttt{readFileContent(filePath: string): Promise<string | null>}:
  richiede il percorso di un file sorgente; restituisce il contenuto testuale del file,
  oppure \texttt{null} se il file non è leggibile. Non propaga eccezioni.
 
  \item \texttt{generateRemediations(filePath, results, fileContent, model):}
  \newline \texttt{Promise<RemediationEntry[]>}: \newline
  richiede il percorso del file, le vulnerabilità rilevate al suo interno, il contenuto
  sorgente e un'istanza del modello LLM; invia tutto al modello, che restituisce un array
  JSON con una remediation per ciascuna vulnerabilità. Ogni remediation è una stringa
  Markdown strutturata in tre sezioni: un esempio della vulnerabilità, le azioni
  correttive o di mitigazione e le conseguenze di una mancata risoluzione. Propaga
  un'eccezione di tipo \texttt{Error} in caso di errore nella chiamata al modello o di
  risposta non parsificabile.
 
  \item \texttt{applyRemediations(vulnerabilities, remediations): void}:
  richiede la lista delle \texttt{VulnerabilityUnit} e le remediation generate; aggiorna
  in-place il campo \texttt{remediation} di ciascuna unità abbinandola tramite
  identificatore. Non solleva eccezioni.
\end{itemize}

\subsubsubsection{DependencyNodeService}
\texttt{DependencyNodeService} implementa l'interfaccia \texttt{INodeScanService} ed è
responsabile della produzione di un inventario completo delle dipendenze della repository
e dell'identificazione delle vulnerabilità note al loro interno, combinando i risultati
di \textit{Syft} e \textit{Grype} con un'elaborazione tramite modello LLM.
\begin{itemize}
  \item \texttt{scan(\{ repoPath \}: \{ repoPath: string \}): Promise<Partial<WorkflowState>>}:
  richiede un percorso valido al repository locale; genera il SBOM tramite \textit{Syft},
  ricerca le vulnerabilità note con \textit{Grype}, traduce le descrizioni in italiano e
  chiede al modello LLM di classificare le dipendenze e produrre un'analisi discorsiva
  delle criticità. Restituisce un aggiornamento parziale del \texttt{WorkflowState}
  contenente il \texttt{DepsReport} completo. Un eventuale fallimento di \textit{Grype}
  viene gestito in modo isolato, producendo un elenco di vulnerabilità vuoto senza
  compromettere il resto dell'analisi. In caso di errore generale, restituisce un report
  con tutti i campi vuoti senza propagare eccezioni.
\end{itemize}
 
\subsubsubsection{DependencyNodeHelper}
\texttt{DependencyNodeHelper} fornisce al \texttt{DependencyNodeService} le operazioni
necessarie all'invocazione degli strumenti di analisi, all'interpretazione dei loro
output e alla classificazione delle dipendenze tramite modello LLM.
\begin{itemize}
  \item \texttt{executeSyft(repoPath: string): Promise<string>}:
  richiede un percorso valido al repository; invoca \textit{Syft} sulla directory e
  restituisce il SBOM (\textit{Software Bill of Materials}) in formato JSON, ovvero
  l'inventario completo di tutte le dipendenze rilevate. Propaga un'eccezione di tipo
  \texttt{Error} al chiamante in caso di fallimento.
 
  \item \texttt{parseSbom(sbomRaw: string): DepsReportUnit[]}:
  richiede il contenuto JSON del SBOM prodotto da \textit{Syft}; restituisce la lista
  delle dipendenze, ciascuna identificata da nome e versione. Propaga un'eccezione di
  tipo \texttt{Error} in caso di JSON non valido.
 
  \item \texttt{executeGrype(sbomRaw: string): Promise<string>}:
  richiede il contenuto JSON del SBOM; lo salva temporaneamente su disco, invoca
  \textit{Grype} per la ricerca di vulnerabilità note e ne restituisce il report in
  formato JSON. Il file temporaneo viene rimosso al termine dell'operazione,
  indipendentemente dall'esito. Propaga un'eccezione di tipo \texttt{Error} al
  chiamante in caso di fallimento.
 
  \item \texttt{parseGrype(grypeRaw: string): DependencyVulnerability[]}:
  richiede il contenuto JSON del report prodotto da \textit{Grype}; restituisce la
  lista delle vulnerabilità rilevate, ciascuna con identificatore CVE, severità,
  descrizione, pacchetto interessato, versione affetta e versione corretta disponibile,
  se presente. Propaga un'eccezione di tipo \texttt{Error} in caso di JSON non valido.
 
  \item \texttt{translateDescriptions(vulnerabilities: DependencyVulnerability[]):}
  \newline \texttt{Promise<DependencyVulnerability[]>}: \newline
  richiede la lista delle vulnerabilità rilevate; invia le descrizioni originali in
  inglese al modello LLM, che le restituisce tradotte in italiano in forma leggermente
  più discorsiva. Restituisce la lista aggiornata con le traduzioni rimappate. In caso
  di errore nella chiamata al modello, restituisce la lista originale invariata senza
  propagare eccezioni.
 
  \item \texttt{analyzeDependencies(repoPath: string, list: DepsReportUnit[],}
  \texttt{vulnerabilities: DependencyVulnerability[]): Promise<DependencyAnalysis>}:
  richiede il percorso del repository, la lista delle dipendenze e le vulnerabilità
  tradotte; legge il \texttt{package.json} e invia tutto al modello LLM, che restituisce
  la separazione delle dipendenze in framework e librerie con le rispettive versioni
  installate, e un'analisi discorsiva in Markdown delle vulnerabilità emerse entro
  un limite di 1\,000 caratteri. In caso di errore, restituisce una
  \texttt{DependencyAnalysis} con tutti i campi vuoti senza propagare eccezioni.
\end{itemize}

\subsubsubsection{DepsReportUnit}\label{sec:DepsReportUnit}
 
La classe \texttt{DepsReportUnit} rappresenta il tipo di trasferimento dati relativo a una singola

\begin{itemize}
  \item \texttt{name: string}: stringa che indica il nome della dipendenza;
 
  \item \texttt{version: string}: stringa che indica la versione della dipendenza rilevata

\end{itemize}
 
\subsubsubsection{DepsVulnerabilityUnit}\label{sec:DepsVulnerabilityUnit}
 
La classe \texttt{DepsVulnerabilityUnit} rappresenta il tipo di trasferimento dati relativo a una
singola vulnerabilità associata a una dipendenza del progetto. Essa espone i seguenti attributi:
 
\begin{itemize}
  \item \texttt{id: string}: stringa che identifica univocamente la vulnerabilità all'interno del
    sistema di scansione;
 
  \item \texttt{severity: string}: stringa che indica il livello di severità della vulnerabilità
    (ad esempio \textit{low}, \textit{medium}, \textit{high} o \textit{critical});
 
  \item \texttt{description: string}: stringa contenente una descrizione testuale della
    vulnerabilità rilevata;
 
  \item \texttt{packageName: string}: stringa che indica il nome del pacchetto affetto dalla
    vulnerabilità;
 
  \item \texttt{packageVersion: string}: stringa che indica la versione del pacchetto affetto dalla
    vulnerabilità;
 
  \item \texttt{fixVersion: string[0..1]}: stringa opzionale che indica la versione del pacchetto
    nella quale la vulnerabilità risulta corretta. Il campo può essere assente qualora non sia
    ancora disponibile una versione risolutiva.
\end{itemize}

\subsubsubsection{SemgrepResult}\label{sec:SemgrepResult}
 
La classe \texttt{SemgrepResult} rappresenta il tipo di trasferimento dati relativo a un singolo
risultato prodotto dallo strumento di analisi statica Semgrep durante la scansione del codice
sorgente. Ogni istanza corrisponde a un'occorrenza di una regola violata all'interno del progetto.
Essa espone i seguenti attributi:
 
\begin{itemize}
  \item \texttt{path: string}: stringa che indica il percorso del file in cui è stata rilevata la
    violazione della regola di analisi statica;
 
  \item \texttt{check\_id: string}: stringa che identifica univocamente la regola Semgrep che ha
    prodotto il risultato, tipicamente nella forma \texttt{<ruleset>.<rule-name>};
 
  \item \texttt{values: Map<string, unknown>}: mappa che associa a ciascuna chiave stringa un
    valore di tipo non determinato a priori, contenente i metadati aggiuntivi restituiti da Semgrep
    per il risultato corrente, quali la posizione nel file, il messaggio descrittivo e la severità.
\end{itemize}
 
\subsubsubsection{RemediationEntry}\label{sec:RemediationEntry}
 
La classe \texttt{RemediationEntry} rappresenta il tipo di trasferimento dati relativo a una
singola indicazione di rimedio associata a una vulnerabilità identificata nel progetto. Essa viene
utilizzata per trasportare le istruzioni di correzione generate per ciascuna vulnerabilità nel
contesto del workflow di analisi. Essa espone i seguenti attributi:
 
\begin{itemize}
  \item \texttt{id: string}: stringa che identifica univocamente la vulnerabilità a cui
    l'indicazione di rimedio si riferisce, in corrispondenza con l'identificativo presente nel
    report delle vulnerabilità;
 
  \item \texttt{remediation: string}: stringa contenente le istruzioni testuali di correzione o
    mitigazione della vulnerabilità identificata.
\end{itemize}
 
\subsubsubsection{RemediationNodeInput}\label{sec:RemediationNodeInput}
 
La classe \texttt{RemediationNodeInput} rappresenta il tipo di trasferimento dati relativo
all'input fornito al nodo del workflow deputato alla generazione delle indicazioni di rimedio per
le vulnerabilità rilevate. Essa espone i seguenti attributi:
 
\begin{itemize}
  \item \texttt{vulnerabilitiesReport: VulnerabilitiesReport[0..1]}: oggetto opzionale di tipo \newline
    \texttt{VulnerabilitiesReport} che contiene il report strutturato delle vulnerabilità da
    elaborare. Il campo può essere assente qualora il report non sia ancora disponibile al momento
    dell'invocazione del nodo;
 
  \item \texttt{vulnerabilitiesReportPath: string[0..1]}: stringa opzionale che indica il percorso
    del file contenente il report grezzo delle vulnerabilità prodotto dallo strumento di scansione
    esterno. Il campo può essere assente qualora il percorso non sia ancora stato determinato o
    qualora il report sia già disponibile in forma strutturata tramite l'attributo precedente.
\end{itemize}




\subsubsubsection{VulnerabilityUnit}\label{sec:VulnerabilityUnit}
 
La classe \texttt{VulnerabilityUnit} rappresenta il tipo di trasferimento dati relativo al dettaglio
di una singola vulnerabilità restituita dal sistema di scansione. Essa espone i seguenti attributi:
 
\begin{itemize}
  \item \texttt{id: string}: stringa che identifica univocamente la vulnerabilità
 
  \item \texttt{path: string}: stringa che indica il percorso del file o del componente in cui la
    vulnerabilità è stata rilevata
 
  \item \texttt{description: string}: stringa contenente una descrizione estesa della natura e
    delle implicazioni della vulnerabilità
 
  \item \texttt{remediation: string}: stringa contenente le indicazioni per la risoluzione o la
    mitigazione della vulnerabilità
 
  \item \texttt{severity: number}: valore numerico che rappresenta il livello di severità della
    vulnerabilità
 
  \item \texttt{impact: string}: stringa che descrive l'impatto potenziale della vulnerabilità
    sul sistema qualora venisse sfruttata
 
  \item \texttt{category: string}: stringa che indica la categoria di appartenenza della
    vulnerabilità (ad esempio \textit{injection}, \textit{broken access control} e simili)
 
  \item \texttt{cwe: string}: stringa che riporta l'identificativo CWE (\textit{Common Weakness
    Enumeration}) associato alla vulnerabilità
 
  \item \texttt{owasp: string[0..*]}: lista di stringhe che riportano i riferimenti alla
    classificazione OWASP associati alla vulnerabilità. Il campo può essere vuoto qualora la
    vulnerabilità non rientri in nessuna categoria OWASP
\end{itemize}

\end{itemize}

  \item \texttt{vulnerabilityAnalysis: string}: stringa contenente l'analisi testuale complessiva

\subsubsubsection{ PartialWorkflowState}\label{sec:PartialWorkflowState}

La classe \texttt{PartialWorkflowState} rappresenta il tipo di trasferimento dati relativo a un
aggiornamento parziale dello stato di un workflow di analisi. A differenza di \texttt{WorkflowState},
tutti i suoi attributi sono opzionali, in quanto essa viene utilizzata per trasmettere esclusivamente
i campi dello stato che hanno subito una modifica in seguito al completamento di una singola fase
della scansione. Essa espone i seguenti attributi:

\begin{itemize}
  \item \texttt{target: Target[0..1]}: oggetto opzionale di tipo \texttt{Target} che, se presente,
    aggiorna il riferimento al repository oggetto della scansione;

  \item \texttt{repoPath: string[0..1]}: stringa opzionale che, se presente, aggiorna il percorso
    locale del repository clonato;

  \item \texttt{startScanTime: Date[0..1]}: valore opzionale di tipo \texttt{Date} che, se
    presente, aggiorna il momento di avvio della scansione;

  \item \texttt{vulnerabilitiesReportPath: Date[0..1]}: valore opzionale che, se presente,
    aggiorna il percorso del file contenente il report grezzo delle vulnerabilità;

  \item \texttt{languages: Language[][0..1]}: lista opzionale di oggetti di tipo
    \texttt{Language} che, se presente, aggiorna l'insieme dei linguaggi di programmazione
    rilevati nel repository;

  \item \texttt{depsReport: DepsReport[0..1]}: oggetto opzionale di tipo \texttt{DepsReport}
    che, se presente, aggiorna il report delle dipendenze del progetto;

  \item \texttt{vulnerabilitiesReport: VulnerabilitiesReport[0..1]}: oggetto opzionale di tipo \newline
    \texttt{VulnerabilitiesReport} che, se presente, aggiorna il report delle vulnerabilità;

  \item \texttt{docsReport: DocsReport[0..1]}: oggetto opzionale di tipo \texttt{DocsReport}
    che, se presente, aggiorna il report della documentazione;

  \item \texttt{testReport: TestReport[0..1]}: oggetto opzionale di tipo \texttt{TestReport}
    che, se presente, aggiorna il report dei test;

  \item \texttt{finalReport: Report[0..1]}: oggetto opzionale di tipo \texttt{Report} che, se
    presente, aggiorna il report finale aggregato del progetto analizzato.
\end{itemize}

\subsubsubsection{ RepositoryConnectionInfo}\label{sec:RepositoryConnectionInfo}

La classe \texttt{RepositoryConnectionInfo} rappresenta il tipo di trasferimento dati relativo
alle informazioni di connessione a un repository GitHub, utilizzato tipicamente in contesti di
verifica parziale della configurazione. Tutti i suoi attributi sono opzionali, in quanto la classe
può essere impiegata per trasmettere anche solo un sottoinsieme delle credenziali disponibili.
Essa espone i seguenti attributi:

\begin{itemize}
  \item \texttt{owner: string[0..1]}: stringa opzionale che indica il nome dell'utente o
    dell'organizzazione proprietaria del repository GitHub;

  \item \texttt{repository: string[0..1]}: stringa opzionale che indica il nome del repository
    GitHub;

  \item \texttt{branch: string[0..1]}: stringa opzionale che indica il nome del branch del
    repository;

  \item \texttt{accessToken: string[0..1]}: stringa opzionale che contiene il token di accesso
    per l'autenticazione verso repository privati.
\end{itemize}

\subsubsubsection{ SendReportInfo}\label{sec:SendReportInfo}

La classe \texttt{SendReportInfo} rappresenta il tipo di trasferimento dati relativo alle
informazioni necessarie per l'invio di un report generato a un destinatario esterno. Essa espone
i seguenti attributi:

\begin{itemize}
  \item \texttt{target: string}: stringa che identifica il destinatario a cui il report deve essere
    inviato, tipicamente un indirizzo email o un identificativo di canale;

  \item \texttt{report: Report}: oggetto di tipo \texttt{Report} che contiene il report completo
    da trasmettere al destinatario;

  \item \texttt{token: string}: stringa contenente il token di autenticazione necessario per
    autorizzare l'operazione di invio verso il servizio di notifica esterno.
\end{itemize}

\subsubsubsection{ SendErrorNotificationInfo}\label{sec:SendErrorNotificationInfo}

La classe \texttt{SendErrorNotificationInfo} rappresenta il tipo di trasferimento dati relativo
alle informazioni necessarie per l'invio di una notifica di errore a un destinatario esterno,
in caso di fallimento del workflow di analisi. Essa espone i seguenti attributi:

\begin{itemize}
  \item \texttt{target: string}: stringa che identifica il destinatario a cui la notifica di errore
    deve essere inviata;

  \item \texttt{token: string}: stringa contenente il token di autenticazione necessario per
    autorizzare l'operazione di invio della notifica verso il servizio esterno.
\end{itemize}
